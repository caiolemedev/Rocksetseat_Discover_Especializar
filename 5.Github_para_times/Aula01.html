Github para times

O que é o Github para times?

Podemos usar o Github para nos auxiliar no trabalho em time.

Criando um repositório

Criando um repositório teste podemos em confgurações adicionarmos os colaboradores do projeto e eles terão acesso ao mesmo repositório. Criamos um arquivo index.html e colocamos algumas informações básicas para termos um conteúdo no mesmo.

Branches

Na sequência, clonamos o repositório para o repositório local e abrimos o VS code local para editar os arquivos. Podemos editar o projeto em nossa branch main e alterarmos o projeto atual (podemos verificar no terminal ou no github desktop). Podemos usar o comando git checkout -b nomeDaNovaBranch e será criada a nova branch e irá nos posicionar nelas, para trocar depois de criada não precisamos do -b. Criamos mais um arquivo e realizamos algumas alterações na aplicação fazendo um commit na sequência. O log de commits é apresentado por branch, o que não estiver na main não aparece. No github podemos selecionar qual branch gostaríamos de trabalhar. Podemos usar branchs alternativas quando a main está no github pages e não queremos quebrar a página.

Merge

Para colocarmos o conteúdo de uma branch na main, colocamos o terminal dentro da main e usamos o comando git merge NomeDaBranchQueQueremosInserir. Assim todas as alterações da branch alteranativa serão inseridas na main. A branch não deixa de existir, permanece no mesmo status que deixamos.

Apagar uma branch

Quando não precisamos mais de uma branch e queremos apagar seu conteúdo, usamos o comando git brach -D branchASerApagada. Não podemos estar dentro da branch para apaga-la. A branch permanece no github e podemos deletar diretamente no site ou usando o comando git push origin --delete branchASerApagada no terminal.

Atualizando branchs

Quando temos a criação de outras branchs que não em nossa máquina local mas estão no github já podemos atualizar o status usando o comando git fetch no terminal ele atualizará na máquina local, mas ele não mostra na lista de branchs. Somente após fazermos um checkout para a branch atualizada é que a branch irá aparecer em nosso terminal. Se deletarmos a branch no github podemos usar o comando git fetch -p para atualizar com a deleções feitas no repositório, mas ainda precisamos usar o git branch -D branchASerApagada para deletar os arquivos também da nossa máquina.

Configuração de branch padrão e proteção da main

Quando trabalhamos em time podemos ter várias branchs e push podem acontecer sem querer e alterar a main. No github conseguimos proteger a main no setting via branch protection rules, podemos marcar que precisamos de um pull request para um merge com aprovação. Adicionamos também que o dono do projeto precisa avaliar antes de ocorrer o merge e que as branchs podem ser excluídas depois do merge. Existem outras configurações, mas não são tão usadas.

Pull Request

Na aba superior do github podemos abrir os detalhes de cada um dos pull requests. Podemos ver quantos commits e de qual branch está vindo o request, bem como os detalhes de cada commit nos detalhes linha a linha. Na aba file changes podemos ver todas as alterações do merge, sem divisão por commit, ou seja, de toda a alteração. Podemos adicionar comentários na linha a ser alterada quando fazemos o review e ao finalizarmos a revisão podemos solicitar alterações para a pessoa que submeteu o request. Após o novo pull request o comentário anterior fica com flag outdated e todas as aprovações são necessárias novamente. Nesta segunda avaliação é melhor utilizar a avaliação por commit. Após as aprovações o github autoriza o botão de merge.

Resolvendo conflito

Quando recebemos o código do github via pull no terminal podemos ter conflitou ou quando alguém alterou alguma linha ao mesmo tempo. Para resolver, podemos apagar as linhas erradas e os códigos do commit que o github coloca e pronto, o conflito está resolvido. Mas não podemos apenas fazer um commit na main após isso, precismos criar uma nova branch para depois fazer o merge.

Criando pull request

Com a branch criada acima podemos criar um pull request via git push origin NomeDaBranchQueQueremosInserir e então no site do github acessamos um novo pull request e inserimos qual a branch que queremos fazer o merge. O github avalia e bloqueia o merge, mesmo após aprovações, de um pull request com conflitos. Adicionamos um comentário com os detalhes e os reviewrs. Podemos adicionar 1 ou mais responsáveis pelo pull request e colocar labels. Podemos filtrar os pull requests por pessoas ou labels. Após alterar o código depois da revisão precisamos solicitar novamente a aprovação, somente o commit não envia automaticamente uma solicitação de aprovação. Na página do pull request podemos restaurar banchs que já excluímos.

Colaborando em projetos públicos

Podemos alterar repositórios públicos via pull requests. Não conseguimos alterar diretamente, podemos clonar mas no caso, criamos um fork que é uma cópia para o nosso github criando um novo repositório antes e depois clonamos em nossa máquina local. Fazemos as alterações que desejamos e adicionamos um commit ao nosso repositório do fork (branch que criamos) e na sequência criamos um pull request para o projeto original. Podemos criar o pull request no repositório original mas precisamos comparar com repositórios com fork (botão azulzinho). A única pessoa que pode aprovar o pull request e realizar o merge é o dono do projeto neste caso. Podemos contribuir somente com reviews em outros repositórios também.

Issues

Issues são pendências que ainda não foram resolvidas ou revisões que não iniciam com pull request. Podemos colocar labels, responsáveis e comentários normalmente, por exemplo a tag good first issue é uma label para melhorias simples no projeto. Após o pull request para resosuloção da issue podemos linkar os dois temas e quando aprovamos e fazemos o merge podemos finalizar a issue (manualmente).

Projects

Para criarmos um projeto dentro do github entramos no repositório e criamos um novo ou adicionamos o repositório em um já existente. No projeto podemos criar as tarefas e quando convertemos elas para issues precisamos linkar um projeto para dar sequência. O github projects utiliza a metodologia do Kanban para manter o tracking das atividades, e caso fecharmos a issue dentro do repositório a tarefa no kanban já migra para o status done. Podemos usar o kanban como um atalho para a discussão da issue. Além de issues podemos linkar as tarefas com PR também
